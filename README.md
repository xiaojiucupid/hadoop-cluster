# 基于 Hadoop 生态的影视智能分析与个性化推荐系统
**融合社交拼团与 AI 智能代理能力的毕业设计全方案**

---

## 一、项目概述

### 1.1 项目名称
**基于 Hadoop 生态的影视智能分析与个性化推荐系统**

### 1.2 项目定位
本项目面向影视数据分析与推荐场景，基于 **Hadoop / Spark / MySQL / Spring Boot / Vue3 / Android** 构建一套完整的影视分析与推荐平台。系统以 **离线大数据分析 + 个性化推荐** 为核心，辅以 **Web 可视化、Android 移动端、拼团交易、AI Agent 智能交互** 等扩展能力，形成从“数据采集—分析建模—推荐服务—用户交互—业务转化”的闭环。

### 1.3 设计目标
1. 完整体现 Hadoop 生态在海量影视数据存储、处理、分析中的工程价值。
2. 实现 Spring Boot 前后端分离架构，支持 Web 端与 Android 双端访问。
3. 在保证毕设核心要求的前提下，融合拼团交易与 AI Agent 能力，提升创新性与答辩表现。
4. 控制实现成本，按照“**先保核心闭环，再做亮点扩展**”的原则推进，确保 10 周开发周期内可落地。

## 二、总体架构设计

系统采用五层架构，自底向上分别为：

### 2.1 数据持久化层
负责系统数据的存储与管理，包括：
- **HDFS**：存储原始影视数据、清洗后数据、离线中间结果；
- **MySQL**：存储用户、影视、评分、收藏、推荐结果、拼团订单等结构化数据；
- **Redis（可选）**：缓存热门影视、推荐列表、拼团活动等热点数据。

### 2.2 大数据计算与分析层
负责离线计算与推荐模型训练：
- **Spark**：ETL 清洗、统计分析、推荐模型训练；
- **Hadoop**：提供分布式存储与资源调度基础；
- **Hive（可选）**：构建离线数仓，提升分析查询效率。

### 2.3 后端服务层
负责业务逻辑与接口封装：
- **Spring Boot**：构建 RESTful API；
- **MyBatis-Plus**：简化持久层开发；
- **JWT**：实现多端统一认证；
- **MySQL**：承载业务核心数据。

### 2.4 智能决策层
负责自然语言交互与工具调度：
- **LLM API**：提供大模型推理能力；
- **LangChain**：实现 Agent 编排与工具调用；
- **Prompt 工程**：约束 Agent 角色与输出边界。

### 2.5 多端交互层
提供用户触达入口：
- **Web 管理与可视化端**；
- **Android 移动端**。

## 四、核心功能设计

## 4.1 必做模块（必须完成）
### 4.1.1 大数据离线分析与预处理
- 基于 Spark 对影视数据进行读取、清洗、格式标准化；
- 空值处理、异常值过滤、字段规整；
- 清洗后数据落地 HDFS；
- 统计分析结果写入 MySQL，供前端展示。

### 4.1.2 多维度影视统计分析
- 题材分布统计；
- 评分区间分布统计；
- 历年影视产量趋势统计；
- TopN 高分影视榜单；
- TopN 热门影视榜单；
- 不同题材平均评分对比。

### 4.1.3 个性化推荐模块
- 基于 Spark MLlib 的 ALS 协同过滤推荐；
- 为每个用户生成 TopN 推荐列表；
- 推荐结果落地 MySQL；
- 支持首页热门推荐、高分推荐、专属推荐等场景。

### 4.1.4 用户管理模块
- 注册、登录、退出；
- 个人信息管理；
- 收藏、评分行为记录；
- JWT 多端统一认证。

### 4.1.5 影视内容管理模块
- 影视列表分页查询；
- 按题材筛选；
- 按名称/演员模糊搜索；
- 影视详情页展示；
- 收藏、评分交互。

### 4.1.6 Web 数据可视化模块
基于 ECharts 完成影视大数据驾驶舱：
- 饼图：题材占比；
- 折线图：历年产量趋势；
- 柱状图：TopN 排行；
- 直方图：评分分布。

### 4.1.7 Android 核心模块
- 启动页、登录注册页；
- 首页、搜索页、详情页；
- 个人中心；
- 收藏与推荐查看。

---

## 4.2 亮点加分模块
### 4.2.1 社交拼团交易引擎
用于增强“推荐 → 转化”的业务闭环。

功能建议：
- 拼团活动创建与管理；
- 开团、参团、满员成功、超时失败状态流转；
- 虚拟积分支付，避免真实支付接入；
- 拼团数据统计与可视化。

设计重点：
- 使用状态机思想管理订单生命周期；
- 不追求复杂支付，重点体现流程闭环与状态控制。

### 4.2.2 影视管家 AI Agent
用于实现自然语言到系统操作的端到端交互。

可支持的工具能力：
- 影视信息查询；
- 个性化推荐查询；
- 拼团创建与辅助操作。

设计重点：
- 通过 Prompt 限定 Agent 仅处理影视和拼团相关任务；
- 通过工具调用实现“问答式操作系统”；
- 不建议本地部署大模型，直接使用可用 API 即可。

## 五、推荐算法设计

### 5.1 混合推荐思路
系统采用 **ALS 协同过滤 + 基于内容的推荐** 的混合方案：

- **ALS 协同过滤**：根据用户评分矩阵训练隐向量，预测用户对未看影视的偏好；
- **基于内容推荐**：根据题材、演员、导演、年份等特征计算相似度，解决冷启动问题。

### 5.2 算法价值
- 适合稀疏评分数据；
- 工程实现成本低；
- 可直接依托 Spark MLlib 落地；
- 论文中可重点说明“原理 + 工程实现 + 效果展示”。

## 八、答辩亮点

### 8.1 叙述主线建议
答辩时可以围绕以下主线展开：

**传统影视推荐系统往往只关注算法本身，缺少数据分析、业务闭环与智能交互能力。本项目基于 Hadoop 生态构建离线分析与推荐系统，在此基础上引入拼团交易与 AI Agent，实现“内容推荐—业务转化—自然语言交互”的完整闭环。**

### 8.2 最值得强调的三点
1. **大数据核心明确**：不是普通 Web 管理系统，而是真正使用 Hadoop / Spark 做离线分析与推荐。
2. **工程完整度高**：Web、Android、后端、数据分析全链路打通。
3. **创新性足够**：拼团 + AI Agent 让项目更有辨识度，答辩更容易讲出亮点。

## 九、避坑建议

### 9.1 环境搭建
- 优先使用伪分布式或 Docker 环境；
- 不建议为了“形式完整”去搭建复杂多节点集群；
- 毕设重点是功能闭环与算法落地，不是运维展示。

### 9.2 算法实现
- 直接使用 Spark MLlib 的 ALS；
- 不建议从零手写复杂推荐算法；
- 论文中重点写原理、流程、效果和工程实现。

### 9.3 AI Agent
- 不建议本地部署大模型；
- 通过 API 调用即可；
- 用 Prompt 控制 Agent 只做影视和拼团相关任务。

### 9.4 拼团模块
- 不接入真实支付；
- 使用虚拟积分或虚拟币；
- 重点体现状态流转与业务闭环。

### 9.5 Android 模块
- 保留核心页面即可；
- 不做复杂动画和过度美化；
- 只要能稳定运行并展示核心流程即可。
